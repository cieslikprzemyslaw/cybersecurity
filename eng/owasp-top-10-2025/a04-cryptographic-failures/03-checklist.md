# A04:2025 - Cryptographic Failures Checklist

## Purpose

Use this checklist when reviewing applications, APIs, cookies, tokens, authentication flows and frontend storage for Cryptographic Failures.

The goal is to identify weak protection of sensitive data and weak token design.

## Core Question

Ask:

> Is sensitive data or a trusted token protected with strong, standard, unpredictable and correctly configured mechanisms?

## 1. Encoding vs Encryption

### Testing Questions

- Is the application using Base64 to hide sensitive data?
- Can a token or cookie be decoded into readable values?
- Does decoding reveal usernames, IDs, roles, hashes, emails, secrets or internal structure?
- Is encoding being confused with encryption?

### Red Flags

```text
Base64(username:password)
Base64(username:hash)
Base64(userId:role)
Base64(email:token)
```

### Safe Notes

Base64 is not encryption.

If the value can be decoded without a secret key, it should not be treated as protected.

## 2. Remember-Me Cookie Review

### Testing Questions

- What is stored in the remember-me cookie?
- Does the cookie decode to readable data?
- Does it contain a username?
- Does it contain a password hash?
- Does it contain a predictable structure?
- Can the token be generated by an attacker?
- Is the token random and high entropy?
- Is the token validated server-side?
- Does the token expire?
- Can the token be revoked?
- Is the token rotated after use or after sensitive actions?

### Bad Pattern

```text
Base64(username:MD5(password))
```

### Better Pattern

```text
random high-entropy token
    -> stored as a hash server-side
    -> linked to user/device/session
    -> expires
    -> revocable
    -> protected with secure cookie flags
```

## 3. Hashing Review

### Testing Questions

- Is MD5 used for password-related values?
- Is SHA1 used for password-related values?
- Are fast hashes used where slow password hashing is needed?
- Are passwords salted?
- Is an adaptive password hashing function used?
- Is the hash exposed to the client?
- Is a password-derived value stored in a cookie or token?

### Red Flags

```text
MD5(password)
SHA1(password)
unsalted password hash
password hash in cookie
password hash in URL
password hash in localStorage
```

### Safer Password Storage

For password storage, use standard password hashing algorithms such as:

- Argon2,
- bcrypt,
- scrypt,
- PBKDF2 where appropriate.

Do not use password hashes as remember-me tokens.

## 4. Cookie Flag Review

### Testing Questions

- Does the cookie need to be readable by JavaScript?
- If not, is `HttpOnly` set?
- Is `Secure` set for HTTPS-only transmission?
- Is `SameSite` appropriate for the application flow?
- Does the cookie have a sensible expiry?
- Does the cookie contain sensitive or predictable data?

### Important Notes

`HttpOnly` does not fix XSS.

However, it prevents JavaScript from reading the cookie through:

```js
document.cookie
```

This can reduce the impact of XSS against session or remember-me cookies.

## 5. Token Randomness and Predictability

### Testing Questions

- Does the token look random?
- Can I predict the token format?
- Can I recreate the token if I know the username?
- Can I generate tokens using a wordlist?
- Does the token include timestamps, IDs or hashes?
- Is the token generated using a cryptographically secure random source?
- Is the token long enough?

### Red Flags

- token contains username,
- token contains user ID,
- token contains role,
- token contains password hash,
- token has a clear delimiter like `username:hash`,
- token can be Base64 decoded into meaningful data,
- token can be brute-forced offline.

## 6. Frontend Storage Review

### Testing Questions

- Are secrets stored in frontend JavaScript?
- Are access tokens stored in localStorage?
- Are long-lived tokens readable by JavaScript?
- Are refresh tokens exposed to JavaScript?
- Are tokens included in URLs?
- Are sensitive values written to browser logs?
- Are tokens included in source maps or bundles?

### Red Flags

```text
localStorage.setItem('token', ...)
window.__CONFIG__.secret
API key in JS bundle
long-lived token in localStorage
password hash in browser storage
```

## 7. Burp Testing Questions

When reviewing cookies or tokens in Burp:

- What happens if I remove the session cookie and keep only remember-me?
- What happens if I Base64 decode the cookie?
- Does the decoded value have a structure?
- Can I change the username and rebuild the cookie?
- Can I use Intruder to generate candidate cookies?
- Do successful and failed token attempts have different response lengths?
- Does the response reveal the authenticated username?

## 8. Burp Intruder Review Pattern

If a cookie follows this pattern:

```text
Base64(username:MD5(password))
```

then Burp Intruder can be used to test candidate passwords by processing each payload:

```text
candidate password
    -> MD5
    -> add prefix: username:
    -> Base64 encode
```

This should not be possible in a secure design.

## 9. Remediation Checklist

For weak remember-me token design:

- Remove password-derived values from cookies.
- Stop using `Base64(username:hash)` style tokens.
- Generate random high-entropy tokens.
- Store only hashed tokens server-side.
- Link tokens to user/session/device where appropriate.
- Add expiry.
- Add revocation.
- Rotate tokens after sensitive events.
- Use `HttpOnly`, `Secure`, and appropriate `SameSite`.
- Invalidate old weak tokens.
- Review logs for possible abuse.

For weak hashing:

- Do not use MD5 or SHA1 for passwords.
- Use Argon2, bcrypt, scrypt or suitable PBKDF2.
- Use salts and work factors.
- Do not expose password hashes to the client.

For XSS + cookie theft risk:

- Fix the XSS.
- Set `HttpOnly` where JavaScript access is not needed.
- Use CSP as defence-in-depth.
- Avoid storing sensitive long-lived tokens in JavaScript-readable locations.

## 10. Developer Red Flags

Be careful when you see:

- Base64 values in cookies,
- remember-me tokens with readable structure,
- MD5 or SHA1 in authentication flows,
- cookie values containing usernames and hashes,
- long-lived tokens without expiry,
- tokens without revocation,
- missing `HttpOnly`,
- token values in URLs,
- secrets in frontend bundles,
- custom crypto helpers,
- comments suggesting “encoded for security”.

## Frontend Engineer Takeaway

Frontend developers do not usually design password hashing or remember-me token storage, but they often see the evidence:

- cookies in DevTools,
- localStorage/sessionStorage,
- token values in requests,
- missing cookie flags,
- Base64-looking values,
- XSS impact on browser-readable tokens.

If a token is readable by JavaScript and has a predictable structure, it deserves review.
