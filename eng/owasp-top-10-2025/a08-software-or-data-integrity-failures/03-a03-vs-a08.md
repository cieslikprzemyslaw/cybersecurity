# A03 versus A08

## A03: Software Supply Chain Failures

A03 examines the broader lifecycle used to select, obtain, build, distribute, and maintain software.

Typical A03 concerns:

- vulnerable or malicious dependencies,
- dependency confusion,
- compromised package accounts,
- unsafe package-management processes,
- weak supplier governance,
- missing dependency inventory or SBOM,
- compromised build systems,
- uncontrolled lifecycle scripts,
- weak update and maintenance processes.

Main question:

> Is the software supply chain secure?

## A08: Software or Data Integrity Failures

A08 examines whether a specific software or data artefact is verified before a trusted component consumes it.

Typical A08 concerns:

- unsigned updates,
- build artefacts deployed without signature or provenance verification,
- scripts loaded from untrusted sources,
- serialized objects accepted from clients,
- client-controlled session properties used for authorization,
- configuration accepted without integrity checks,
- old signed data replayed outside its intended context.

Main question:

> Can this specific artefact or data be trusted, and has it retained its integrity?

## Practical distinction

```text
A03: process, suppliers, dependencies, build and distribution lifecycle
A08: verification of a specific artefact or data item at a trust boundary
```

## Examples

### Malicious npm dependency

Start with A03 because the problem concerns dependency selection and the software supply chain.

A08 may also apply if the build process accepts or deploys an artefact without verifying its source, signature, or provenance.

### Modified session cookie

Start with A08 because the backend accepts altered client-controlled data without sufficient integrity verification.

The impact may also be an access-control failure if the altered value grants new permissions.

### Unsigned application update

A08 is the direct category because a specific update is installed without proving authenticity and integrity.

A03 may be relevant when reviewing the larger release and distribution process.

## Avoid incorrect mapping

Do not force every vulnerable package into A08.

Do not ignore A08 simply because the artefact passed through CI/CD.

Classification should reflect the root failure being reviewed:

- broad supply-chain process failure -> A03,
- missing integrity/authenticity verification of a consumed artefact -> A08.
