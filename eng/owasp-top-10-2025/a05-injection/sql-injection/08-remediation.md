# SQL Injection Remediation

## Primary fix

The primary fix for SQL Injection is:

```text
Prepared statements / parameterized queries
```

This keeps SQL structure separate from user-controlled values.

## Vulnerable concept

```text
SQL query text + user input = dangerous
```

Example vulnerable idea:

```php
$sql = "SELECT * FROM books WHERE book_name = '$book_name'";
```

If `book_name` contains SQL syntax, it may change the query.

## Secure concept

```text
SQL structure is fixed.
User input is passed separately as a parameter value.
```

Example concept:

```text
SELECT * FROM books WHERE book_name = ?
parameter value = user-provided book name
```

The database treats the input as data, not executable SQL syntax.

## Additional layers

Prepared statements should be supported by other controls:

- allowlist values where possible,
- validate type and format,
- use safe ORM patterns,
- avoid unsafe raw queries,
- avoid verbose database errors in production,
- use least-privilege database users,
- monitor suspicious query behaviour,
- add regression tests.

## GET vs POST

Changing a request from GET to POST does not fix SQL Injection.

GET vs POST changes where the input is sent, not whether it is safe.

```text
The security boundary is how the backend builds and executes the SQL query.
```

POST or JSON body may make API design cleaner, but it does not prevent SQLi by itself.

## Developer takeaway

Do not trust input just because it comes from:

- a URL generated by the application,
- a dropdown,
- a cookie,
- a hidden field,
- a frontend-controlled value,
- a CMS-managed field.

The backend must handle all input safely.
