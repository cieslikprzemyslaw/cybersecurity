# A03 kontra A08

## A03: Software Supply Chain Failures

A03 analizuje szerszy cykl wyboru, pozyskania, budowania, dystrybucji i utrzymywania software.

Typowe problemy A03:

- podatne lub złośliwe zależności,
- dependency confusion,
- przejęte konta maintainerów,
- niebezpieczne procesy package management,
- słabe zarządzanie dostawcami,
- brak inventory lub SBOM,
- przejęte systemy build,
- niekontrolowane lifecycle scripts,
- słabe procesy aktualizacji i utrzymania.

Główne pytanie:

> Czy software supply chain jest bezpieczny?

## A08: Software or Data Integrity Failures

A08 analizuje, czy konkretny software albo artefakt danych jest weryfikowany przed użyciem przez zaufany komponent.

Typowe problemy A08:

- niepodpisane aktualizacje,
- artefakty builda wdrażane bez weryfikacji podpisu lub pochodzenia,
- skrypty ładowane z niezaufanych źródeł,
- serializowane obiekty przyjmowane od klientów,
- właściwości sesji kontrolowane przez klienta używane do autoryzacji,
- konfiguracja bez kontroli integralności,
- stare podpisane dane odtwarzane poza przeznaczonym kontekstem.

Główne pytanie:

> Czy temu konkretnemu artefaktowi lub danym można zaufać i czy zachowały integralność?

## Praktyczne rozróżnienie

```text
A03: proces, dostawcy, zależności, build i cykl dystrybucji
A08: weryfikacja konkretnego artefaktu lub danych na granicy zaufania
```

## Przykłady

### Złośliwa paczka npm

Zacząłbym od A03, ponieważ problem dotyczy wyboru zależności i software supply chain.

A08 może mieć dodatkowe zastosowanie, jeśli build akceptuje lub wdraża artefakt bez sprawdzenia źródła, podpisu lub provenance.

### Zmienione session cookie

Zacząłbym od A08, ponieważ backend akceptuje zmienione dane kontrolowane przez klienta bez wystarczającej weryfikacji integralności.

Wpływem może być również access-control failure, jeśli zmieniona wartość daje nowe uprawnienia.

### Niepodpisana aktualizacja aplikacji

A08 jest bezpośrednią kategorią, ponieważ konkretny update jest instalowany bez potwierdzenia autentyczności i integralności.

A03 może być istotne przy przeglądzie całego procesu release i dystrybucji.

## Unikanie błędnego mapowania

Nie należy wciskać każdej podatnej paczki do A08.

Nie należy ignorować A08 tylko dlatego, że artefakt przeszedł przez CI/CD.

Klasyfikacja powinna opisywać główną przyczynę:

- szeroki problem procesu supply chain -> A03,
- brak weryfikacji integralności lub autentyczności konsumowanego artefaktu -> A08.
